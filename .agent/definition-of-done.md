# Definition of done

Development-phase gate. Short on purpose — see `PROJECT_RULES.md` for the principle and `.agent/workflow.md` for the procedure.

The point is to catch real regressions, not hypothetical ones. If a check below doesn't apply to your change, skip it and say so in the report.

---

## Minimum gate (every change)

- [ ] **`pytest -q` is green** — or only flags pre-existing issues unrelated to your change. Note them in the report.
- [ ] **`API_CONTRACT.md` matches the code** — if the route, schema, status code, header, or error envelope changed, the contract document changed in the same commit. If it didn't change, neither did the contract.
- [ ] **No obvious regression in adjacent flows** that share `app/storage/*` or another common module with your change. "Obvious" = something a reviewer would catch by looking, not exhaustive coverage.
- [ ] **Module boundaries respected** — `app/storage/*` did not learn about HTTP, `app/routers/*` did not open a file, `app/segment/*` is still self-contained. (See `PROJECT_RULES.md §3`.)
- [ ] **No frontend implementation details** — code comments, specs, and changelogs do not name vision components, pages, or click flows. (See `PROJECT_RULES.md §0`.)
- [ ] **Commit authored as `paikku <jungi_@naver.com>`** — if it slipped through as `Claude <noreply@anthropic.com>`, self-correct right away with `--amend` or `git filter-branch --env-filter` + `git push --force-with-lease`. Every commit must set the four env vars `GIT_AUTHOR_NAME` / `GIT_AUTHOR_EMAIL` / `GIT_COMMITTER_NAME` / `GIT_COMMITTER_EMAIL` (procedure: `.agent/workflow.md` §6).
- [ ] **Commit / PR / MR text in Korean** — subject + body, no English conventional-commit prefixes (`docs:`, `feat:`, etc.), no Claude session links / `Co-authored-by: Claude` / "generated by" footers. Docs (`__spec__.md`, `__changelog__.md`, `.agent/*.md`, `API_CONTRACT.md`, code comments) stay in English; only commit / PR / MR text is Korean.

That's the floor. Most changes finish here.

## Add to the gate when the change warrants it

- [ ] **Live `curl` against `uvicorn`** — when the change touches a route, header, status code, or error envelope. Don't trust unit tests alone for contract surface; one round-trip through the full middleware chain (CORS, exception handlers, lifespan state) catches things like wrong Origin echo back.
- [ ] **Round-trip with a real fixture** — when the change touches `app/normalize.py` (decode-required AVI / MP4 remux fixture) or `app/segment/*` (real JPEG, model weights present).
- [ ] **One error path walked** in addition to the happy path — when the change touches input validation (Pydantic bodies, Form parsing), upload size limits, queue-cap behavior, or timeout handling.
- [ ] **Storage layout check** — when the change touches `app/storage/*`, write to a fresh tmp `STORAGE_ROOT` and confirm the on-disk shape matches what `__spec__.md` says (index files, atomic-write tmp residue, cascade cleanup).

## Documentation (only when it earns its keep)

See `.agent/workflow.md` §5 for when these apply. The default is to skip them.

- [ ] **`API_CONTRACT.md` updated** — only if a route, schema, status code, header, or error code changed.
- [ ] **`__spec__.md` updated** — only if a documented internal behavior, invariant, or pitfall changed.
- [ ] **`__changelog__.md` updated** — only if your change shifts a non-obvious *current* design decision in the area. Replace / rewrite the relevant entry rather than appending history.
- [ ] **`REFACTORING.md`** — bumped only if you observed a deferred refactor candidate that the team should track.
- [ ] **`PROJECT_RULES.md`, `CLAUDE.md`, `AGENTS.md`, `.agent/*.md` are unchanged** unless the task was specifically about agent procedure or principles.

---

## Honesty rule

If you skipped a check, **say so in the task report** with a one-line reason. A clean run with no manual verification is *not* done — it's a code change of unknown quality. Be honest about what was and wasn't verified.

That's the only thing that doesn't loosen in development mode. The team (and the next agent) needs to know the actual state of the work, not the optimistic state.
